# Active Client Proof Pipeline Gates

Created: 2026-05-13
Purpose: one cross-client gate for turning active client work into proof, referrals, content, or sales assets without overclaiming, exposing private data, or asking before acceptance.

## Canonical Order

Do not publish, pitch, ask for referrals, or reuse proof until the relevant gate is true or explicitly marked blocked.

1. **Acceptance / payment / scope gate**
   - Deliverable boundary is clear.
   - Client has accepted the output or a demo/dry run is recorded.
   - Payment, approval, deposit, or pause status is known.
   - Open issues have owner/date.
   - If client-side data is dirty or incomplete, build work stays paused until the clean input contract is accepted.

2. **Evidence capture gate**
   - Redacted screenshot, run log, output sample, acceptance wording, or demo note exists.
   - Evidence is saved in the client folder, not only remembered from chat.
   - Claims are labeled as one of: exact quote, paraphrase, measured metric, qualitative observation, or unverified/TODO.
   - Missing proof is treated as a blocker, not a blank to fill creatively.

3. **Permission / anonymization gate**
   - Default posture: anonymized client type + workflow category.
   - Naming the client, contact, private workflow, screenshots, metrics, or exact quotes requires explicit permission from JT/client evidence.
   - Redact: client names if unapproved, tenant/property addresses, internal reports, account/policy numbers, deal notes, credentials/tokens, private file paths, financial details, and search/acquisition strategy.

4. **Referral ask gate**
   - Referral asks happen only after acceptance + proof safety are clean, unless the ask is explicitly framed as general feedback/validation with no client claim.
   - Ask for 2–3 fit-qualified intros, not broad blasts.
   - Fit criteria must be included so the referrer can qualify before sending names.
   - Relationship tracker must be current before the ask: update `memory/networking/contacts.md` with last touch, gate status, privacy constraints, MC alignment, and next action.
   - Internal docs/scripts are not relationship touches. If the real-world outcome is unknown, the only allowed next action is refresh/logging.
   - Family/network asks are validation-only until named buyer pain is confirmed; do not frame them as sales referrals or requests for favors/access.

### Relationship Refresh Gate

Before any relationship can be used for an ask, `memory/networking/contacts.md` must show:
- Real-world last touch date and outcome, not just an internal asset/update date.
- Ask type: refresh, validation-only, referral, proof/testimonial, or event follow-up.
- Exact blocker if ask is not allowed.
- JT-owned refresh action when source-of-truth context lives outside files.
- Matching MC task or explicit `no task needed` rationale.

5. **Distribution gate**
   - Content, website snippets, decks, and outreach inserts can use anonymized proof only.
   - Do not imply an unfinished module is live.
   - Do not claim ROI, hours saved, number of records, or production usage unless measured or confirmed.
   - Posted URLs/replies become proof assets only after logged in the relevant proof/content file.

## Quick Stop Rule

If a proof statement requires the words "probably," "should," "I think," or "we can say," stop. Capture evidence or label it as unverified instead.
