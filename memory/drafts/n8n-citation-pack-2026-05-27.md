## n8n Community Draft

**Title:** How I scope an SMB n8n workflow before building it

One pattern I keep seeing with small businesses: the automation request usually starts too late.

They ask for "an n8n workflow" after the process has already become painful.

The better starting point is not the tool. It is the handoff.

Before building, I map five things:

1. **Trigger:** What exact event starts the work?

Example: invoice received, job marked complete, form submitted, spreadsheet row added.

2. **Owner:** Who is responsible when the workflow cannot decide?

Every useful automation still needs a human exception owner.

3. **Source of truth:** Which system wins when two records disagree?

For SMBs, this is usually the real problem. The spreadsheet, inbox, portal, and CRM all tell slightly different stories.

4. **Failure path:** What should happen when the API is down, the field is blank, or the model is unsure?

If there is no failure path, the workflow is not ready for production.

5. **Business result:** What behavior changes if this works?

"Send an email automatically" is not a result.

"Client gets a same-day update without the owner calling three people" is a result.

That scope usually makes the build smaller.

It also makes the workflow easier to maintain, because the automation is wrapped around the business process instead of forcing the business process to adapt to the automation.

Curious how other n8n builders handle this: do you scope around the trigger first, or the exception path first?
