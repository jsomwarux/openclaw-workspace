A client should not need one report for leads, one invoice for API spend, and one run log to know whether automation was worth it.

I have started treating paid research agents the same way I treat tenant, vendor, or owner-facing workflows:

- source input
- allowed action
- approval state
- spend cap
- evidence log
- stop condition

The useful part is not that the agent can pull more data.

The useful part is that the operator can see when the next pull stops being worth it.

That matters for paid APIs, lead research, app distribution, and client operations. If the workflow keeps acting after the budget signal turns red, it is not ready.

The build standard is simple:

The agent can draft, rank, and recommend.

The operator still sees the inputs, budget, approval state, and proof before anything expensive or customer-facing happens.
