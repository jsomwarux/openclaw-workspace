A client dashboard build I am scoping right now does not start with the model.

A dashboard request, a playbook agent, and a QA agent can become three separate projects if the acceptance record is unclear.

That is where implementation scope gets messy.

The useful artifact is the record around the work.

What source did the agent read?

What output did it generate?

What did QA check?

Who reviewed the risky part?

What stayed manual?

What changed in the dashboard, workflow, or client handoff after review?

I am seeing this more clearly while scoping current implementation work.

The model choice matters, but it is not the control surface.

The control surface is the record that lets the business say:

- this source was allowed
- this action was in scope
- this output passed review
- this exception was held
- this owner accepted the result

That is the difference between AI help and production work.

If the only proof is "the agent ran," the project is not finished.

The business needs to know what changed, who accepted it, and where the evidence lives.
