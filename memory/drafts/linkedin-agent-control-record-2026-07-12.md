One property report, one dashboard queue, and one tool call should not become three different stories.

The workflow is not production-ready when it can draft the update.

It is ready when the business can inspect the record around the action.

What source did it read?

Which system did it request access to?

Was the action allowed, denied, or routed to a reviewer?

Where did the output land?

What happens if the result is wrong?

That is the part the first demo usually skips.

The workflow might draft the rent follow-up.

It might prep the dashboard.

It might update the queue.

But the buyer still needs an operating record they can defend later.

For ops-heavy teams, I would rather start with one boring control row than a bigger agent demo:

- source export
- requested action
- permission boundary
- approval owner
- held exception
- audit path
- rollback rule

Once that exists, the agent stops feeling like a black box.

It starts looking like workflow infrastructure.

The draft is not the product.

The control record is.
