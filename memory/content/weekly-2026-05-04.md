## Wednesday — Case Study: Maintenance Request Triage
**Pillar:** Case Study | **Day:** Wednesday 2026-05-06

A maintenance inbox gets messy because every request arrives with the same weight.

A leaking pipe. A noisy neighbor. A broken appliance. A vague “something is wrong” message from a tenant.

If the property manager has to read every request, classify urgency, decide who owns it, and write the response, the system is not triage. It is manual sorting with a nicer inbox.

While building a maintenance triage workflow, the useful piece was the first decision.

Claude reads the request, classifies the issue type, assigns urgency, drafts the tenant response, and routes the work to the right next step. Emergencies get surfaced. Routine requests keep moving. Ambiguous ones stay visible for a human.

The build became a reusable n8n template for property management intake, not a one-off demo.

The point was not faster replies.

The point was making sure the dangerous request stopped looking like every other message.

## Thursday — Agentforce Proof Point
**Pillar:** Agentforce Proof Point | **Day:** Thursday 2026-05-07

A good AI agent is not the one that answers the most questions.

It is the one that refuses the right work.

That showed up while building four Agentforce agents. The useful design decisions were not just prompts and actions. They were boundaries: what the agent can resolve, what it should ask for, and what has to stay with a person.

A customer-facing agent with no stopping point becomes a risk surface.

A narrower agent with clean escalation becomes operational infrastructure.

## Friday — Tactical Insight: Buyer Channel Enrichment
**Pillar:** Tactical Insight | **Day:** Friday 2026-05-08

A prospect list is not a pipeline until the buyer channel is real.

Finding the right company is the easy part. Finding the operator with authority, a reachable channel, and a reason to care is the work that decides whether the draft is sendable.

Half-enriched prospects create hidden maintenance debt. They look close enough to keep around, but not clear enough to contact.

Those belong in a warm-up queue until authority and channel are both confirmed.

## Sunday — Behind the Build: Backend Voice Agents
**Pillar:** Behind the Build | **Day:** Sunday 2026-05-10

Voice demos get expensive when every interaction needs a frontend first.

That changes the shape of what is worth building.

A backend voice relay means the useful test can happen earlier: can the agent understand the request, call the right tool, and respond in a way the operator would trust?

For consulting demos, that matters.

You can prove the workflow before polishing the interface.

## Backup 1 — Friday Swap
**Best fit:** Friday | **Slug:** workflow-before-agent

An AI agent should not be your first design decision.

The first decision is where the work changes hands.

If the handoff is vague, the agent will inherit the mess. If the handoff is clear, the agent has a job: read the input, make the decision, route the next step, escalate the edge case.

Better agents start with narrower workflows.
