## Wednesday - Case Study
**Date:** 2026-05-13  
**Pillar:** Case Study  
**Source:** recent-builds.md, content-voice.md Proof Points, current-efforts.md  
**From seed:** false

Property teams do not need another dashboard if someone still has to hunt for the missing listings by hand.

The expensive part is not opening StreetEasy.

It is remembering to check it, applying the same criteria every time, copying the matching listings into a spreadsheet, and doing it again two weeks later.

That is the kind of workflow I look for first.

For a real estate client, I built a scraper that runs every 14 days, checks listings against custom criteria, and sends the matches into a Google Sheet.

The outcome was simple: 4 hours of manual search work removed every cycle.

Not a giant AI transformation.

A repeatable search process turned into a scheduled system.

That is usually where AI implementation should start.

Find the recurring judgment call.

Define the criteria.

Move the repeatable part out of someone's calendar.

**Source:** current-efforts.md, jtsomwaru.com AI Operations Diagnostic reposition  
**From seed:** false

The first AI diagnostic question should not be “what can we automate?”

It should be “what should stay manual until the data is clean?”

That is not a cautious answer. It is the fastest path to a real build.

If the ledger is messy, rent delinquency automation becomes a confidence problem. If the vendor list is stale, maintenance routing becomes noise. If nobody owns the exception, the workflow just moves confusion into a faster channel.

Good implementation starts by choosing the first workflow that can survive contact with real data.
