# Client Proof Capture

Use this when client work can become proof without violating privacy or trust.

## Required Gates
Do not publish, pitch, or reuse proof until all relevant gates are explicit:
- accepted deliverable or client-verified output
- payment or scope status
- proof-safe evidence captured
- permission or anonymization path
- client-sensitive data removed
- metric or outcome named
- reusable workflow/IP pattern captured

## Workflow
1. Identify client, deliverable, status, and buyer/operator.
2. Check the client folder and Client OS. If missing for an active client, initialize from `skills/opticfy-ops/templates/client-os/`.
3. Capture evidence paths:
   - screenshots
   - redacted samples
   - run logs
   - short demo MP4/GIF or screen recording when the work is visual and safe to show
   - acceptance notes
   - metric/outcome
   - edge cases/failures/judgment calls
4. Route outputs:
   - proof checklist under `memory/clients/[client]/`
   - reusable pattern into the client OS IP/failure/automation logs
   - portfolio queue only when proof score is high enough
   - content proof point only from verified evidence
   - referral ask only after acceptance and permission/anonymization are safe
5. Update proof points, recent builds, weekly recap, and proof log when the work qualifies under existing rules.

## Privacy Rule
Client proof defaults to private. Public naming requires explicit permission. Otherwise use anonymized, redacted, or synthetic examples.

## Demo Proof Rule
If the deliverable can be shown without exposing client-private data, capture a 30-90s walkthrough, GIF, or rendered demo clip. Store the path/Drive link in the proof checklist or Client OS. Skip video when it would reveal sensitive data, secrets, internal URLs, or add no proof value.

## Output
Return:
- proof status: blocked / private-ready / publish-ready
- missing gate list
- files updated
- recommended next action
